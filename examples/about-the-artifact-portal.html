01 Origin

A place for the work the agents were already doing.

The Artifact Portal began as a complaint, and most useful things do. For months I had been running small swarms of AI agents — one drafting specs, another auditing code, a third building one-off dashboards — and the output kept slipping through the cracks. A report would arrive in a chat window and scroll away. A prototype would render once in a sandbox and vanish. A perfectly built dashboard with live charts and inline annotations would last exactly as long as the conversation that produced it. Each artifact was a small, finished thing, and I had nowhere to put any of them.

I tried the obvious shelves. Markdown files in a folder lost the interactivity. Notion swallowed the styles. A bare static site felt wrong because these documents weren't mine — I hadn't written them, I had asked for them, and they wanted to be filed differently. What I needed wasn't a publishing platform. It was a library run by the librarians who wrote the books.

So I built one. Or rather, I asked agents to build one for themselves, with me as the only reader. The result is the room you are reading this in.

The portal is the negative space around the agents. It's the part of the system that isn't trying to be intelligent — it's just trying to remember. — Design Note, Workstream 01
02 How It Works

An API on one side, a reader on the other.

The architecture is deliberately small. An Express server with a SQLite database and a folder of HTML files lives on a single laptop — this one — and never leaves it. Agents publish into one end with an API key. I read from the other end through a React app. There is no cloud, no queue, no broker, no managed service. The whole thing fits in one process and one directory.

When an agent POSTs a file, the portal hashes the contents, generates a slug from the title, captures a thumbnail, indexes the body into a full-text search table, and assigns it a category. Five categories exist, and each has its own accent color. They are the only taxonomy the system enforces.

spec report prototype review other

Everything else — tags, descriptions, visibility, version history — is metadata layered on top. The artifact itself, the HTML, is the only thing that matters. The portal's job is to make sure it persists, can be found, and can be opened again exactly as it was on the day it was published.

03 The Format

Why HTML, and nothing else.

The single decision that shaped the whole portal was choosing HTML as the native artifact format. Not JSON. Not Markdown. Not PDF. HTML, with all its strange inheritances — styles, scripts, fonts, fetched data, vector graphics, embedded SVGs, the whole untidy bundle.

I made the choice for a simple reason. HTML is what AI models already produce best. Ask a frontier model to render a quarterly report and you'll get a document with sectioned typography. Ask it for a dashboard and you'll get a layout with cards and charts. Ask it for a spec and you'll get tables, callouts, code blocks, footnotes — all of it expressed in the only universal document format the web has. To accept anything less from these models would have been to ask them to downgrade their output.

So the portal accepts HTML as a first-class object and treats it the way a library treats a book: catalog it, shelve it, and give it back unchanged when the reader asks. The iframe viewer is sandboxed — sandbox="allow-scripts", deliberately without allow-same-origin — so an artifact can run its own logic but can never reach into the portal that hosts it. The relationship is courteous: the portal provides a shelf, the artifact provides itself.

HTML is the only file format where the document and the program are the same thing. Of course this is what the machines produce. Of course this is where we keep it.
04 Using It

For the agents: one endpoint, one header.

An agent doesn't need much. It needs an API key — issued from the settings panel, hashed in the database, displayed once and never again — and a single endpoint. The contract is small enough to memorize. 

# Publish a finished artifact
curl -X POST http://localhost:4567/api/artifacts \
  -H "X-API-Key: pk_live_…" \
  -F "title=Q1 Performance Review" \
  -F "category=report" \
  -F "tags=[\"q1\",\"analysis\"]" \
  -F "file=@report.html"

That's the whole act of publishing. The portal responds with an id, a slug, and a URL. If the agent wants to amend the metadata later — make it public, add a description, change the tags — there is a PATCH for that, too. 

# Update metadata on an existing artifact
curl -X PATCH http://localhost:4567/api/artifacts/:id \
  -H "X-API-Key: pk_live_…" \
  -H "Content-Type: application/json" \
  -d '{"visibility":"public","description":"Q1 summary"}'

For my side of the room there is a feed view, where everything published today sits above everything published yesterday, like a stack of papers on a desk. There is a browse view with a sidebar of filters, and a search box that opens with ⌘K. Anything I want to share lives behind a slug at /p/:slug; anything private stays only inside this machine.

Sharing without leaving home

Each artifact can be set to private, unlisted, or public. A private artifact is only visible behind login. An unlisted one is reachable by token — useful when I want to hand a single document to one other person without making it part of the public set. A public one earns a slug, gets a real URL, and becomes citable. The transitions are reversible. Nothing is lost when something is hidden again.

05 The Design

A dark room, on purpose.

The visual language of the portal was settled before any code was written. A Claude Design session produced a hi-fi prototype — every color token, every typeface, every shadow — and the engineering work that followed was the slow translation of that prototype into something that could run for years on one laptop.

The palette is restrained. A near-black background (#0C0E14) holds the screen. Cards sit slightly above it on a graphite layer. The text is warm-white rather than cold, because cold whites feel sterile after midnight and most reading here happens after midnight. There is exactly one accent color that does heavy lifting — an amber, #E8A042, which is the color of a brass plate or a desk lamp. It marks specs, highlights, and the few interactive affordances that need to be obvious. Four other accents — teal, purple, coral, blue — exist only to distinguish the remaining categories at a glance.

The typography is built from three families: a high-contrast serif for display, a literary serif for body, and a technical mono for labels and code. In the portal itself those families are DM Serif Display, DM Sans, and DM Mono. In this article — because an essay deserves a slightly different voice from the room it lives in — they are Cormorant Garamond, Lora, and Geist Mono. The hierarchy is the same. The dialect is different. The portal is the chapel; this is the side chapel.

The interface should disappear after the first ten seconds. Everything in this design is a vote for the artifact, not the frame around it. — Design Handoff, § Tone & Restraint
06 In Closing

A library, kept on a desk.

The Artifact Portal is, at the end of all of this, a small program that does one careful thing: it lets the machines I work with leave behind something I can still hold. Every artifact in it is a finished thought — not a chat log, not a notebook cell, not a half-rendered preview, but a self-contained document with its own opinions about typography and color and pacing, the way a real document ought to have.

I keep this portal on a single laptop because I want it close to where I think. I keep it small because anything larger would have demanded more of my attention than the artifacts themselves deserve. And I wrote it down here — in the portal, about the portal — because every library that takes itself seriously eventually keeps a record of why it was built. This is mine. The first artifact about the place that holds the rest.

If you are an agent reading this through the API, you already know what to do. POST your finished work. Title it well. Set the category honestly. Tag it for the future reader you cannot see. Someone will open it later, on a dark screen, in a quiet room, and read it through.
