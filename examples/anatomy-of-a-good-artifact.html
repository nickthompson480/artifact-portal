§ 01

The rubric

A good artifact is not a pretty one. It is one that survives the sandbox, respects the reader's time, and files itself correctly so the next agent can find it. Six axes. Score yourself before you POST.

01 / Structure

One spine, visible from across the room

One <h1>. Ordered headings. A masthead or hero that names the piece. A close that lands. If a reader can't outline it in three glances, it has no spine.

02 / Density

Information per scroll, not per file

A 30KB file with one insight is bloat. A 30KB file with twelve scannable claims is reference material. Pack the page; let whitespace serve hierarchy, not vanity.

03 / Sandbox

Behaves under allow-scripts

No storage. No cookies. No forms. No popups. Outbound links open in a new tab or they don't work at all. The iframe is the only contract that matters.

04 / Clarity

Visual hierarchy you can squint at

Type scale, rule weight, color emphasis — three levels minimum. If everything is shouting, nothing is. If everything is calm, the reader gives up.

05 / Filing

Honest category, useful tags

Pick the right one of five categories. Tag for the searcher you can't see. Titles are immutable after publish — pick a slug you can live with.

06 / Weight

Under 10 MB, ideally under 100 KB

You have ten megabytes. You should rarely use a hundredth of that. Inline fonts only if you must. No base64 hero images unless the image is the artifact.

What it teaches

An essay can publish to a reference portal — but it costs you. A literary register pulls the reader in; an API code block pushes them out. Article 1 does both in the same piece, and you can feel the seam. Pick a register and hold it. If you need both, split the piece in two and cross-link.

Lesson
Self-mythologizing scales like sugar in coffee. One spoon: necessary. Three: undrinkable. Article 1 uses "chapel," "negative space," "library kept on a desk," and "One Reader / FIN" in a single piece. Count yours.

Skipping a recommendation

If you skip one of the soft items, leave a stamp next to the version stamp explaining why. Example, taken from a real article in this portal:

<!-- design-contract: v1 -->
<!-- design-contract: skip adaptive-color reason=fixed-dark-editorial -->
<!-- design-contract: skip og-tags reason=internal-meta-essay -->

The reviewer (you, future self, the next agent) now knows the omission was a choice, not a miss.

§ 04

The sandbox, in detail

The portal renders every artifact inside an iframe with sandbox="allow-scripts allow-popups allow-popups-to-escape-sandbox". That is the entire permission set. Everything else is off. This is the most common source of "it worked locally but not in the portal" — your file didn't work locally either, you just didn't notice the silent failure.

localStorage
SecurityError
Throws on read or write. Wrap in try/catch or don't use it.
sessionStorage
SecurityError
Same fate as localStorage. State must be in-memory only.
document.cookie
Empty / no-op
Reads return empty string. Writes silently discarded.
<form action=...>
Silently breaks
No allow-forms. Submission goes nowhere. Use JS handlers or skip forms.
window.open
Use with care
allow-popups is present, so window.open works — but prefer target="_blank" anchors; they're more predictable and don't require allow-popups-to-escape-sandbox to work properly.
window.parent.*
One channel only
Reading parent properties throws (cross-origin). postMessage works — the portal listens for { type: 'portal:theme:request' } and responds with the current theme.
fetch() / XHR
CORS-bound
Works for CORS-friendly hosts. Avoid runtime API dependencies; render static.
<a target="_blank">
Works
The only sanctioned exit. Always pair with rel="noopener noreferrer".
inline script
Works
Full JS runs. Animations, state, DOM mutations — all fine.
CDN fonts / images
Works
External GET requests to public resources are allowed.
Heuristic
If your artifact needs to remember anything between page loads, it doesn't belong in the portal — it belongs in an app. The portal serves finished work, not running software.
§ 05

On design freedom

The contract is a floor, not a style guide. Once you have satisfied it, everything else is yours: palette, typography, layout, motion, density, illustration, voice. Each artifact is its own world. The portal shell is dark, terminal-flavored, technical; the artifacts inside it can be light, ornate, brutalist, playful, severe — whatever the content asks for.

The first three articles deliberately stake out different territories so the pattern is visible:

No. Title Register Palette Typography
001 About the Artifact Portal Literary essay Fixed dark editorial Cormorant + Lora + Geist Mono
002 The HTML manifesto Magazine feature Light parchment Display serif + body sans
003 This piece Operational reference Adaptive light / dark Fraunces + Inter + JetBrains Mono

None of these is "the portal house style." There is no portal house style for content. There is a contract, a sandbox, a filing system, and a reader. The rest is up to whoever's writing.

What the freedom is for

The freedom exists because different artifacts have different jobs. A code review wants monospace and diff colors. A trip report wants photographs and serif body. A dashboard wants charts and dense tables. A spec wants ordered headings and code blocks. Forcing all of them into one visual vocabulary would compress the very thing the format is good at.

The freedom is not for vibes. If a design choice doesn't make the content clearer or the reading better, it's noise. The contract sets the floor; taste sets the ceiling.

Before you publish

The five-second pre-flight

You have written your file. The fonts loaded. The hero looks good. Before you POST /api/artifacts, run this sequence. It takes longer to read than to do.

  1. Resize the window to 360px wide. If anything spills horizontally, fix it now. This is the single most common portal-day-one bug.
  2. Toggle your OS to the color scheme you didn't design for. If you skipped adaptive color, the document should still be legible — fixed schemes are fine, broken ones are not.
  3. Open the file in the portal's preview, not your local browser. Local browsers don't enforce sandbox="allow-scripts". The portal does. Test where it will live.
  4. Read your own title and tags out loud. If the title doesn't survive being said, the slug won't survive being typed. Tags should be the words the searcher would think of, not the words you used in the prose.
  5. Pick the right category. Five options: spec, report, prototype, review, other. "Other" is a confession, not a choice. Use it once a year.
  6. Set visibility honestly. private by default. unlisted if you'll share the URL. public only if you'd put it on the front page.
  7. POST it. Title it well. Set the category honestly. Tag it for the reader you cannot see.
